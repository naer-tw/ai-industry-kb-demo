AI產業知識庫capability

自動bug修復

一句話定義（這個可交付物是什麼）

給定一段含 bug 的函式、一條會失敗的測試（用來定位並界定正確行為），AI 產生一個修補，使其能通過該失敗測試、同時不破壞既有（既有測試所涵蓋的）行為；最後由人類 review 後合併。相對於「整張工單／跨多檔 PR」（見 GitHub-issue端到端PR），本任務是 AI 程式修復裡定位已給、範圍最窄、規格最明確的子集：失敗測試把「錯在哪、應如何」直接框好，AI 不需自行探索整個倉庫。

難度評分（綜合 3/10：技術 3、落地 4）— 理由(引 benchmark 與部署)

  • 技術成熟度 = 3（可用、需人工把關）：本任務把 AI 程式修復中最難的「定位」外部化——失敗測試＝可執行的正確行為規格。研究證實「有 bug 重現測試」會顯著拉高修復率（見實證一的 cogeneration 論文），而本任務直接把該測試當輸入給定。在此設定下，頂尖模型於 SWE-bench Verified（單 issue、測試導向）達 88.6~95.5%（2026-06）。但純函式級 APR 的「老實」數字仍受 patch 複雜度壓制：單行 plausible@1 僅 45%、多 hunk 降到 27.5%（2025；且 plausible≠correct）。綜合落在「3 可用但需人工把關、已有產品」。
  • 落地難度 = 4（已商品化、仍須人 review）：產品成熟且大規模——Cursor Bugbot 每月處理 200 萬+ PR、覆蓋 11 萬+ 倉庫，其 Autofix 自動修補有 35%+ 直接併入原 PR（2026-02 起，見實證二）。但「不破壞既有行為」是已知硬傷：overfitting／plausible patch 問題——修補通過了給定測試卻沒真正修好、或破壞測試未涵蓋的行為——是 APR 數十年的老問題，故契約明確保留「使用者審核合併」這道人為防線，落地分數略高於技術。
  • 綜合 = 3：能做、有規模化產品、有真實合併數據，且因「失敗測試已給＋單函式」屬最易子集，故比 difficulty 5 的端到端 PR 低 2 分；但 overfitting 與「保證不破壞既有行為」仍需人 review，未到 1-2「成熟可靠廣泛無人值守」之列。
  • 查證重評：3/4/3（與原評一致，無分歧。2026-06-26 反查：SWE-bench Verified 88.6~95.5%@2026-06-18 屬實；plausible@1 45.06/34.37/27.50 出自來源 3 而非原註之來源 2，已更正並改標 plausible@1；D4C 180/437、Cursor 2M+ PR/月＋35%+ Autofix 併入＋~80% 修復率均屬實）。

實證一：Benchmark（測項＋分數＋日期＋來源）

  • SWE-bench Verified（500 題人工驗證、單 issue 解決、測試導向）：2026-06-18 Claude Mythos 5 95.5%、Claude Fable 5 95.0%、Claude Opus 4.8 88.6%、GPT-5.5 88.7%。此為「有測試框定、單點修復」的能力上緣。[來源 1]
  • patch 複雜度梯度（Generalizable APR 實證，2025）：plausible@1 單行 45.06% → 單 hunk 34.37% → 多 hunk 27.50%。注意是 plausible（通過測試、未驗證真正修好）而非 correct；顯示一旦超出「單點小修」，連「貌似通過」率都明顯下滑。[來源 3]
  • Defects4J 單函式 bug（LLM-APR survey 彙整）：D4C（prompt 內嵌失敗測試、10 sample/bug，無需完美定位）修好 437 個單函式 bug 中的 180 個（≈41%），較前 SOTA 高約 10%；ChatGPT 修好 337 中 162（約 $0.42/bug）。[來源 2、3]
  • 失敗測試的價值（agentic 修復的 bug 重現測試，2026-01）：有 bug 重現測試者，跨 benchmark 修復率顯著高於無；本任務正是把該測試當輸入給定，等於先天站在較高起點。[來源 5]
  • 判讀：Verified 的 88~95% 對應「測試已給的單點修復」，與本任務契約最貼；但需扣掉 overfitting 與資料污染（OpenAI 內部稽核指出部分 Verified 題目曾出現在訓練資料，模型可逐字重現 gold patch），純函式級「老實」數字更接近 27~45%。

實證二：真實部署/產品案例（誰用、規模、成效）

  • Cursor Bugbot / Autofix：每月處理 200 萬+ PR、覆蓋 11 萬+ 倉庫；70%+ 的旗標在 merge 前被修掉。Autofix（2026-02 上線）會在隔離 VM 起一個 Cloud Agent 去測試、產修補並提案，35%+ 的 Autofix 提案被直接併入原 PR；2026 年比較研究中 Cursor 在「修復類任務」的接受率達 80.4%。[來源 4]
  • GitHub Copilot coding agent / Codex 等：均提供「指派失敗測試或 bug→自動修補並開 PR」流程，企業端仍以分支保護＋人工核可把關；端到端情境見 GitHub-issue端到端PR。
  • 成效結論：在「範圍明確＋失敗測試已給」的修復類任務上，AI 修補已能達 35~80% 被人類採用／合併，屬真實規模化部署；但「直接 merge 不需改」仍非常態，人類 review 是合約內建環節。

現況與邊界（AI 目前做得到/做不到）

  • 做得到：單函式、單點（單行／單 hunk）邏輯 bug；失敗測試已界定正確行為時的修補；通過給定測試、且多數情況通過既有測試套件；在好的代理（Cursor Autofix 等）上高比例被工程師採用。
  • 做不到／仍脆弱：跨多檔、需理解 bug 成因脈絡的 regression（context-aware 增強才有 1.6 倍提升，且傳統 APR 完全失敗）；測試未涵蓋的隱性行為——overfitting／plausible patch 會「通過測試卻沒真修好」；難以自證「不破壞既有行為」，需既有測試夠完整＋人 review 兜底。
  • 瓶頸：patch 正確性驗證（plausible≠correct）、測試套件的不完整性、複雜度上升時的可靠度衰減、以及責任歸屬——故「使用者審核合併」是必要、而非可選。
