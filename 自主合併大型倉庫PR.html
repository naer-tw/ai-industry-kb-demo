AI產業知識庫capability

自主合併大型倉庫PR

一句話定義（這個可交付物是什麼）

給 AI 一個「大型生產倉庫」的開放式 feature ticket（自然語言、需求未完全規格化），由 AI 全自動定位程式碼、實作、過測試、開 PR，且在「無任何人工審查或介入」下穩定合併到主幹，一次合併成功率 >90%。重點不在「能不能寫出 patch」，而在「不需人看就敢進 main 且九成以上一次過」。

難度評分（綜合 8/10：技術 7、落地 9）— 理由(引 benchmark 與部署)

  • 技術 7：模型「寫得出可用 patch」已從前沿邁向可用，但最貼近本任務的大型/商業倉庫 benchmark 仍遠低於 90%。SWE-bench Pro（1,865 題、41 個真實/專有倉庫、平均改 107 行跨 4 個檔）標準化榜首 GPT-5.4 僅 59.1%，廠商自報 Claude Opus 4.8 也只有 69.2%（2026-06-18）。較簡單的 SWE-bench Verified（500 題 Python）已達 Opus 4.8 88.6%（2026-05-28），但那是「規格清楚、單倉庫」的測項，最貼近本任務的 Pro 仍只有 60-69%。技術成熟但「大型倉庫一次過 90%」尚未達標，故給 7 而非更低。
  • 落地 9：本任務的真正障礙在「無人工介入 + >90% 一次合併」這個信任門檻。生產數據顯示自主 PR 合併率收斂在約 67-68%（Devin 67%、微軟 .NET 團隊 Copilot 878 個 PR 合併 535、67.9%），且所有主流系統都強制人工審查、agent 不可自行 merge。完全無人審查的「dark factory」目前僅用於重構/測試/相依升級等窄域、低風險任務，開放式 feature ticket 仍須人把關。可靠性、責任歸屬、安全與監管信任未解，故落地給 9。
  • 綜合 8：技術接近、落地遠未到。整體屬「實驗室/早期、可靠性低或落地障礙大」區間偏上，定 8（與 P1 初判一致）。
  • 查證重評：8/7/9（與原評一致，無 divergence。所有 benchmark／生產數據經 2026-06-26 反查屬實；唯 SWE-bench Verified 已由 Opus 4.6 80.8% 更新為 Opus 4.8 88.6%@2026-05-28，但屬單倉庫規格清楚測項，不影響「大型開放式倉庫無人介入 >90%」未達標的結論，故分數不變）。

實證一：Benchmark（測項＋分數＋日期＋來源）

  • SWE-bench Pro（最貼近本任務：大型真實/專有倉庫、長程任務、過倉庫測試套件）
    • 標準化（Scale SEAL 公開集）：GPT-5.4(xHigh) 59.1%；Claude Opus 4.6(thinking) 51.9%；Gemini 3.1 Pro 46.1%（2026-06-18，morphllm 榜）。
    • 廠商自報（自有 scaffolding）：Claude Opus 4.8 69.2%（同日，數字高於標準化方法）。
  • SWE-bench Verified（較簡單，僅 Python、12 倉庫 500 題）：2026-05-28 Claude Opus 4.8 已達 88.6%（Opus 4.6 80.8%、Gemini 3.1 Pro 80.6%）；早期 2026 Claude 4 Sonnet 77.2%、GPT-5 74.9%。注意此測項為單倉庫、規格清楚，與本任務（大型/多倉、開放式）落差大，不可外推。
  • 解讀：即便最強模型在「最像本任務」的測項也僅約 60-69%，距 >90% 一次合併目標有 20-30 個百分點落差；且 benchmark 是「過測試」而非「無人審查進 main」，後者門檻更高。

實證二：真實部署/產品案例（誰用、規模、成效）

  • Cognition Devin：PR 合併率由前一年 34% 升至 67%（2026）；官方定位仍為「初級工程師等級」，要求 scoped access、PR 交付、CI gate、強制人工 review。
  • 微軟 .NET 團隊 + GitHub Copilot coding agent：2025-05～2026-03 開出 878 個 PR、合併 535 個（67.9%），回退率 0.6%（優於人工 0.8%）。但 agent 不能自行核可或 merge，永遠交人審查。
  • Stripe Minions：每週產出逾千個 AI PR，仍由人審查。
  • 整體：2026 約 18% 已合併 PR 的主要/協作作者為 coding agent；但「無人審查直接進 main」（dark factory）僅在重構、寫測試、文件、相依升級等窄域、可控失敗範圍內落地。尚無針對「大型倉庫開放式 feature ticket、無人介入、>90% 一次合併」的規模化部署證據。

現況與邊界（AI 目前做得到/做不到）

  • 做得到：在中小型或規格清楚的任務上自動開 PR、過 CI；窄域任務（重構/測試/相依升級）可近乎自主；生產合併率約 2/3。
  • 做不到（本任務的缺口）：① 大型倉庫長程任務一次過率僅約 60-69%（benchmark）/67-68%（生產），未達 90%；② 「無人介入」未被任何主流系統允許於開放式 feature ——皆強制人工 review，agent 不可自 merge；③ 開放式（需求模糊）ticket 的需求釐清、跨檔副作用、安全/邊界案例仍是主要失敗模式。
  • 結論：技術正快速逼近「能寫」，但「無人敢放手 + 九成一次過」屬尚未跨越的信任與可靠性門檻。
