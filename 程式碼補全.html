程式碼補全

一句話定義（這個可交付物是什麼）

給定游標前（prefix）與游標後（suffix）的程式碼上下文，AI 即時產出游標處應插入的下一段程式碼（單行到多行），以灰字內聯（ghost text）顯示，使用者按 Tab 接受、否則忽略繼續打字。技術上即 fill-in-the-middle（FIM，填空式）內聯補全。這是生成式 AI 商品化最徹底、單次失誤成本最低（一鍵忽略即可）的能力之一——人在迴路天生內建（每筆建議都由開發者即時決定收/退）。

難度評分（綜合 2/10：技術 2、落地 2）— 理由（引 benchmark 與部署）

  • 技術 2（很成熟）：FIM 已是專門訓練目標，主流碼模型（Qwen2.5/3-Coder、StarCoder2、DeepSeek-Coder、GitHub 自訓模型）均原生支援 prefix/suffix 感知。GitHub 2025-10-23 上線的自訓補全模型，相較舊後端「採納率+12%、被接受並保留字元+20%、延遲-35%、token/秒吞吐 3x」（來源 1）。子任務在公開 benchmark（HumanEval-Infilling 單行填空、CrossCodeEval 跨檔）已是標準測項（來源 4、5）。未給 1，是因為跨檔/長倉庫語境與多行邏輯正確性仍非滿分。
  • 落地 2（已廣泛可靠部署）：億級規模商品化、邊際成本極低、失誤可一鍵忽略，符合錨定量表「1–2 已成熟商品化可靠廣泛部署」。落地分比「短文摘要(3)」更低，因本任務的人工把關是「天生內建於互動」（每筆 Tab 都是即時驗收），不像摘要需另設抽查流程。
  • 綜合 2：技術成熟、部署廣、容錯設計優；唯一拉高難度的是補全內容仍可能含 bug／安全瑕疵，需開發者判讀後接受，故非 1。

實證一：Benchmark（測項＋分數＋日期＋來源）

  • 真實採納率（最貼近本契約的線上指標）：GitHub 引述客戶研究約 30% 建議被採納；企業實測（ZoomInfo）平均建議採納 33%；CACM 研究行採納約 21–24%；Java 開發者最高約 61%（語言差異大）；被接受的碼約 88% 留存於最終提交。GitHub 另宣稱開發者約 46% 的程式碼由 Copilot 產出、受控實驗顯示完成 JS HTTP server 任務快約 55%（皆為 GitHub 自家公布數據，非來源 2 TechCrunch 該篇所載）。
  • 新自訓模型對舊後端的相對提升（GitHub Blog，2025-10-23，來源 1）：採納率+12%、保留字元+20%、延遲-35%、吞吐 3x；以近 1,000 萬倉庫、600+ 語言去重語料中期訓練，並以合成微調修正「游標錯位插入、prefix 重複、suffix 覆寫」等 FIM 常見缺陷。
  • 離線 FIM 測項（來源 4、5）：HumanEval-Infilling＝單行填空（Python/Java/JS，pass@1）；CrossCodeEval/CCLongEval＝跨檔補全（Exact Match、Edit Similarity）；RepoEval＝倉庫級多檔依賴。Qwen2.5-Coder 技術報告（來源 5，2024-09）將 FIM 列為核心評測，於 HumanEval-Infilling 與跨檔 EM/ES 對同級開源碼模型取得領先；本契約多落在「單行～數行、單檔近鄰語境」，屬這些測項最易端。
  • 口徑提醒：採納率是「建議被接受比例」，非「補全正確率」；30% 採納＝多數建議被忽略仍屬正常互動（容錯設計），不代表 70% 是錯的。

實證二：真實部署/產品案例（誰用、規模、成效）

  • GitHub Copilot（來源 2，TechCrunch 2025-07-30）：累計逾 2,000 萬用戶、90% 財星百大採用、5 萬+ 組織付費；2025 年 4–7 月三個月新增 500 萬用戶。受控實驗顯示完成 JS HTTP server 任務快約 55%。內聯補全為其原生核心功能。
  • Cursor（來源 3，2026）：>100 萬日活、約 200 萬用戶（逾百萬付費）；ARR 2025-06 破 $500M、2025-11 破 $1B、2026-02 破 $2B；64% 財星五百大採用、企業端每日逾 1 億行程式碼。其 Tab 補全曾為主力（2025-03 Tab 用戶為 Agent 用戶 2.5 倍），現雖被 Agent 反超（Agent 為 Tab 2 倍），但 Tab 補全仍是每日 95% 用戶最高頻互動。
  • 其他：Amazon Q Developer（前 CodeWhisperer）、JetBrains AI、Codeium/Windsurf、Tabnine 等皆以內聯補全為基礎功能，VS Code 系生態幾乎標配。
  • 成效定性：補全是「分鐘級高頻、低風險」場景，採納率雖僅約 30%，但因每筆只需一個 Tab、忽略零成本，整體淨產出明顯為正，是 AI 寫碼工具最先被廣泛接受的入口能力。

現況與邊界（AI 目前做得到/做不到）

  • 做得到：常見樣板碼、樣式重複碼、單檔近鄰語境的單行到數行補全；FIM 已能正確對齊游標、不覆寫 suffix；億級部署、延遲低於數百毫秒、邊際成本極低。
  • 需把關／做不太穩：(1) 跨多檔、深層倉庫依賴的長段補全正確率下滑（CrossCodeEval/RepoEval 即為此設）；(2) 補全可能含邏輯 bug 或安全瑕疵（ACM 研究指 Copilot 建議正確性有限，來源 6），須開發者判讀；(3) 冷僻語言、私有 API、快速變動框架的補全品質較弱。
  • 結論：以本任務（游標前後上下文→單行至多行補全，Tab 收/退）而言，AI 已可靠廣泛交付、屬最成熟商品化能力之一；風險集中在「不審視即接受多行補全」導致的潛在 bug/安全問題，而互動設計（一鍵忽略＋即時驗收）天然緩解此風險。
