AI產業知識庫capability

完整2D遊戲原始碼

一句話定義（這個可交付物是什麼）

給定一段自然語言功能需求（例如「做一個有跳躍、敵人、計分與通關旗的橫向卷軸平台遊戲」），AI 一次（或少量回合）產出整套可執行原始碼，且必須同時滿足三個硬條件：(1) 遊戲邏輯正確（碰撞、狀態機、勝負判定無誤）、(2) 執行時可穩定達 60fps、(3) 玩家能從頭玩到尾、真正完整通關。這是「能生成跑得起來的遊戲片段」與「能交付一份可上架、可玩完的成品代碼」之間的分水嶺。

難度評分（綜合 7/10：技術 7、落地 6）— 理由(引 benchmark 與部署)

  • 技術成熟度 = 7（實驗室/早期、可靠性低）：在最對口的 benchmark V-GameGym（2,219 個 pygame 遊戲、70 個模型）上，頂尖模型整體解決率僅 o3 49.2%、Gemini-2.5-Pro 47.5%、GPT-5 47.1%（2025-09，見實證一）——意即即使放寬到「品質達標即算過」，最強模型一次性也只有約一半遊戲合格。更關鍵：所有樣本「執行成功率 100%」（代碼跑得起來），但品質分布呈長尾，多數落在 Poor/Fair——即使是品質表現最佳的 GPT-5，其 2,219 次產出中被評為 Excellent（80-100 分）者也僅 83 個（3.7%；其餘 Good 288、Fair 676、Poor 1,172，見 Table 2）。本任務的「正確邏輯＋60fps＋可完整通關」正是 Excellent 等級的要求，而非「跑得起來」，故技術分給 7。
  • 落地難度 = 6（前沿落地但需重度人工把關）：實務上「AI 寫整套遊戲代碼」已被廣泛使用——Steam 上逾 7,300 款遊戲揭露使用 GenAI，且 Valve 自 2026-01 明定「代碼類 AI 工具不需揭露」（見實證二）。但這些幾乎都是「人類開發者用 Claude Code/Cursor 多回合迭代＋親自測試把關」，而非「一句需求一次產出可通關成品」。「達 60fps」與「可完整通關」需實機測試與調優，AI 無法自我保證，仍須人持續驗證，故落地分數 6。
  • 綜合 = 7：落在量表「7-8 實驗室/早期、可靠性低或落地障礙大」的下緣。簡單小遊戲（貪食蛇、打磚塊）AI 已能一次寫好；但「任意功能需求＋三項硬條件全達標」目前不可靠，且品質達標率僅約一半、頂級品質率 <4%，故綜合給 7 而非 5-6。

實證一：Benchmark（測項＋分數＋日期＋來源）

  • V-GameGym（Visual Game Generation for Code LLMs，arXiv 2509.20136，2025-09）：2,219 個高品質 pygame 樣本、100 個主題叢集、評測 70 個模型。以「代碼＋靜態截圖＋動態錄影」多模態綜合評分。[來源 1、2]
    • 整體解決率（達品質門檻者 / 2,219）：o3 49.2%（1,092）、Gemini-2.5-Pro 47.5%（1,054）、GPT-5 47.1%（1,047）、最佳開源 Qwen3-235B-A22B-Thinking 46.8%（1,039，總榜第 4）。
    • 品質長尾：全部樣本「執行成功率 100%」，但多數落在 Poor/Fair；即使最佳模型 GPT-5，2,219 次產出中達 Excellent（80-100）者僅 83 個（3.7%，Table 2）。論文明示「高品質遊戲代碼生成仍具挑戰」。
  • 判讀：benchmark 的「解決」是品質達標，不等於本任務要求的「正確邏輯＋60fps＋完整通關」三條件同時成立。V-GameGym 未把「穩定 60fps」與「人類可實際通關」設為獨立硬指標，故本任務真實達成率應低於 47~49% 這個上限值——這也是評等級為 B、信心設 low 的原因。

實證二：真實部署/產品案例（誰用、規模、成效）

  • 尚無「一句需求一次產出可通關成品」的規模化部署；現況是「人＋AI 多回合協作寫遊戲」的工作流已普及：
    • Steam：截至 2026 約 7,818 款遊戲揭露使用 GenAI（約佔 Steam 庫 7%），2025 年新發行遊戲近 20% 揭露使用；但揭露多集中於美術素材（約 60%）。Valve 於 2026-01 改版規則，明定協助寫程式/除錯的 AI 工具「不需揭露」——代碼類 AI 已是日常工具。[來源 4]
    • 開發者實作案例：以 Claude Code 約 20 分鐘做出可玩的復古小遊戲為公開教學範例；但流程是人類給需求、AI 產代碼、人類實機測試與多輪修正，而非單次保證 60fps＋通關。[來源 3]
  • 成效結論：「AI 輔助寫遊戲代碼」已落地且規模可觀；但「輸入需求→自動交付正確邏輯＋60fps＋可完整通關的成品代碼」尚無規模化、可靠的端到端部署，仍以人為測試與調優的最後防線。

現況與邊界（AI 目前做得到/做不到）

  • 做得到：規則明確的經典小遊戲（貪食蛇、打磚塊、俄羅斯方塊、單畫面平台跳躍）多能一次或少回合產出可玩代碼；作為人類開發者的起手架構與素材生成助手，已是主流工作流。
  • 做不到/仍脆弱：任意複雜需求一次到位的「正確邏輯＋穩定 60fps＋可完整通關」三條件同時達標——頂尖模型品質達標率僅約一半、Excellent 率 <4%；效能（60fps）與可通關性需實機驗證，AI 無法自我保證；較長關卡設計、平衡性、邊界 bug、跨關卡狀態一致性仍易出錯。
  • 瓶頸：可靠度（同一需求能否穩定產出三條件全達標的成品）、效能調優需執行期回饋、可玩性/平衡屬隱性設計判斷、以及缺乏把「60fps＋人類可通關」設為硬指標的成熟自動驗證 benchmark。
