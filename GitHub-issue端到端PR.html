AI產業知識庫capability

GitHub-issue端到端PR

一句話定義（這個可交付物是什麼）

給定一個 GitHub issue（自然語言描述的缺陷或功能需求）與整個倉庫，AI 代理自主探索程式碼、定位需修改的檔案、跨多檔改寫實作、補上對應測試、跑過 CI，最後開出一個可直接合併的 PR；人類僅需 review 並按 merge，不需自己動手改 code。這是「能寫單檔 patch」與「能取代一名工程師完成一張工單」之間的分水嶺。

難度評分（綜合 5/10：技術 4、落地 6）— 理由(引 benchmark 與部署)

  • 技術成熟度 = 4（偏低，已可用）：單一 issue→patch 在 SWE-bench Verified 上已達 88.6~95.5%（2026-06，見實證一），技術主幹成熟。但本任務的硬要求是「多檔＋補測試＋真正可合併」，對應 SWE-bench Pro（平均觸及 4.1 檔、私有倉庫）僅 59.1%（標準化）／69.2~80.3%（廠商報告）。Verified→Pro 普遍掉 15~35 分，顯示「乾淨單檔」與「端到端整張工單」之間仍有實質差距，故給 4 而非更低。
  • 落地難度 = 6（前沿落地但需人工把關）：產品已 GA（GitHub Copilot coding agent、Devin、Codex），但「使用者 review 後直接 merge」的可靠度高度分歧——生產環境純自主 PR 合併率，Copilot 僅 6.2%、Devin 21.6%，而 Codex 達 82.6%（見實證二）。意即：能不能「直接 merge」極度依賴代理品質與任務範圍，且分支保護、人工核可、安全掃描在企業端仍是強制關卡，無法真正無人值守，故落地分數高於技術分數。
  • 綜合 = 5：屬量表「5-6 前沿可做、可靠性/成本仍挑戰、部分落地」的下緣。能做、有產品、有真實合併數據，但「review 即 merge 不必改」尚非常態，仍需資深工程師把關。

實證一：Benchmark（測項＋分數＋日期＋來源）

  • SWE-bench Verified（單 issue 解決率，500 題人工驗證）：2026-06-18 Claude Mythos 5 居首 95.5%、Claude Fable 5 95.0%、Claude Opus 4.8 88.6%；GPT-5.5 88.7%（2026-04-23）。[來源 1、4]
  • SWE-bench Pro（多檔、平均觸及 4.1 檔、含私有/copyleft 倉庫、抗記憶化）：2026-06 標準化榜首 GPT-5.4(xHigh) 59.1%；廠商報告 Claude Opus 4.8 69.2%、Claude Mythos 5 80.3%。私有子集更低：Opus 4.1 由 22.7%→17.8%、GPT-5 由 23.1%→14.9%。[來源 2、5]
  • 判讀：本任務要求「多檔＋補測試」，正確對標是 Pro 而非 Verified。Pro 的 59~69% 才是「端到端整張工單」的真實上限；Verified 的 88%+ 會高估能力。
  • 查證註（2026-06-26）：上述數字經反查均屬實且為 2026-06-18 榜面最新值。Mythos 5 廠商報告 Pro=80.3%（BenchLM）；另 Morph 榜記 Mythos「Preview」版為 77.8% 並標示因出口管制暫停——版本/scaffold 不同，故「69.2~80.3%（廠商報告）」區間成立。

實證二：真實部署/產品案例（誰用、規模、成效）

  • GitHub Copilot coding agent（2026 GA）：指派一張 issue 即自主規劃、開分支與 PR、寫 code、跑測試、自我 review 並完成安全掃描，最後請人 review。企業端分支保護仍生效，代理 PR 須人工核可才能跑 CI/CD。[來源 4]
  • 大規模生產數據（arXiv 2606.22711，2026-06）：分析 5 個代理（Copilot/Devin/Cursor/Codex/Claude Code）共 33,596 個 PR（2024-12～2025-07）。純自主 PR 整體合併率 79.8%、協作式 53.8%；但此聚合數受 Codex 占比 64.9% 拉抬（Simpson 悖論）。逐代理純自主合併率：Codex 82.6%、Cursor 62.7%、Claude Code 55.1%、Devin 21.6%、Copilot 6.2%。Cognition 另報 Devin 在「範圍明確任務」上達 67% 合併率。[來源 3]
  • 成效結論：在「範圍清楚、好的代理」上，端到端 PR 已能達 6~8 成直接合併；但跨代理、跨真實情境的可靠度落差極大，無法宣稱「普遍可 review 即 merge」。

現況與邊界（AI 目前做得到/做不到）

  • 做得到：單檔/小範圍 bug fix、明確規格的功能、補測試、開出格式正確且 CI 通過的 PR；在頂尖代理（Codex 等）上多檔 PR 已能高比例被直接合併。
  • 做不到/仍脆弱：大型/陌生/私有倉庫的長程多檔改動（Pro 私有集 <20%）；架構級重構、跨服務變更、需要產品判斷或隱性慣例的工單；「無人值守自動 merge」——企業仍以分支保護＋人工核可＋安全掃描攔截，純自主合併率在多數代理偏低（Copilot 6.2%）。
  • 瓶頸：可靠度（同一張工單能否穩定產出可合併解）、長程任務 token/成本、對未見程式碼的泛化、以及信任與責任歸屬。技術曲線陡升，但「review 即 merge」仍以人為最後防線。
